01 Executive Summary

Situation. McDermott Will & Schulte is deploying the Anthropic Claude AI platform — comprising Claude Cowork for knowledge workers and Claude Code for engineering and AI analyst teams — as the foundation for an AI-augmented delivery model. The firm currently operates an all-Microsoft Azure stack with React/TypeScript, .NET, SQL Server, and SharePoint Online. This document defines how to extend that stack to support agentic AI workflows, secure MCP integrations, and an automated delivery pipeline.

Core recommendation. The existing stack is well-suited to Claude AI development and requires targeted additions rather than wholesale change. The three highest-priority additions are: Python (for AI pipeline work), Node.js/TypeScript MCP servers (for custom connector development), and Azure Container Apps (for isolated, scalable MCP hosting). Everything else builds on infrastructure the firm already owns and operates.

Architecture in one sentence. Users interact with Claude through role-appropriate surfaces (Cowork for attorneys, Claude Code for developers); all MCP connector traffic routes through a governed gateway on Azure Container Apps that enforces allowlists, manages credentials from Azure Key Vault, and logs all tool calls to Microsoft Sentinel; and every Claude-integrated application is built and deployed through a Claude Code-augmented Azure DevOps pipeline.

Implementation approach. Four phases over 12 months, beginning with identity and governance foundations and the iManage Cloud MCP connector as the reference implementation, expanding to all practice groups and a full skills library by month 12.

KEY PRINCIPLE
Nothing in this architecture creates new privileged access paths. The iManage connector passes the user's own OAuth token — Claude can only see what that attorney already has access to in iManage. The MCP gateway enforces this and logs every access. This is the existing access control model extended to an AI client.
02 Current Stack Assessment

The table below assesses each current technology against its suitability for Claude AI development, MCP connector work, and agentic workflows. The overall finding is positive — the firm's Azure-centric, TypeScript-led stack aligns well with Anthropic's tooling. The gaps are specific and addressable. Click any row to expand its full rationale.

Existing React + TypeScript Front-end

Assessment: Native strength for Claude Code. MCP clients are written in TypeScript. The existing SPFx and standalone React experience transfers directly to Claude-integrated front-ends.

Recommendation: Keep and extend. TypeScript is also the primary language for custom MCP connector development (alongside Node.js runtime), so this investment compounds.

Existing .NET (C#) Middle tier / APIs

Assessment: Well supported by Claude Code. An official Anthropic .NET SDK exists. Claude Code generates idiomatic C# accurately and understands Entity Framework, LINQ, and Azure SDK patterns.

Recommendation: Keep and extend. The .NET API layer becomes the MCP adapter for internal SQL Server data — exposing business logic as MCP tools without ever opening SQL Server to direct connector access.

Existing SQL Server Relational backend

Assessment: Claude Code generates accurate T-SQL. No native MCP connector exists for SQL Server, which is the correct situation — direct database MCP connectors are a security anti-pattern for production systems.

Recommendation: Keep, but never expose directly via MCP. All SQL Server access from Claude routes through the .NET API layer, which enforces business rules and access controls before any query reaches the database.

Extend with caveats SharePoint Online / SPFx Hosting

Assessment: SharePoint Online is excellent for portal-style front-ends and will continue to be. However it imposes fundamental constraints that make it unsuitable for agentic AI workloads: no persistent server-side processes, no WebSocket support for streaming responses, sandboxed execution preventing direct iManage API calls, and no MCP server hosting capability.

Recommendation: Keep for portals — dashboards, document viewers, search interfaces. Move all server-side AI logic, MCP servers, and background agents to Azure Container Apps. SharePoint becomes the front-end surface; Azure is the back-end. This is an evolution of the existing pattern, not a replacement.

CAUTION
Resist the instinct to default to SharePoint for every new application. Establish a clear decision rule by end of Phase 1: if the application needs Claude to do background work, stream responses, or maintain server-side state, it goes to Azure Container Apps — not SharePoint.
Existing Azure (platform) Infrastructure

Assessment: Excellent alignment. The Anthropic API works natively from Azure compute with no special configuration. Azure AD, Key Vault, Sentinel, Container Apps, and AI Search all integrate cleanly into the Claude platform architecture without requiring third-party tooling.

Recommendation: Strong foundation — expand with Container Apps for MCP hosting and Azure AI Search for RAG. The existing Azure investment compounds rather than being displaced.

Add now Node.js / TypeScript MCP servers

Assessment: The fastest path to custom MCP connectors. Anthropic's MCP SDK has a TypeScript-first implementation. Claude Code itself runs on Node. The team already knows TypeScript — this is an extension of existing skills, not a new language.

Recommendation: Primary language for all MCP connector development. The iManage, M365, and Azure DevOps connectors should all be Node.js/TypeScript.

Add now Python AI pipelines

Assessment: The highest-priority addition to the stack, even for a .NET shop. Anthropic's Python SDK is more mature and heavily documented than the .NET SDK. The broader AI/ML ecosystem — sentence-transformers, PyMuPDF, LangChain, LlamaIndex — exists almost exclusively in Python.

Recommendation: Required for AI pipeline work — document chunking, embedding generation, RAG orchestration. At least one Python-proficient developer must be in place by the start of Phase 2. Claude Code generates high-quality Python, lowering the bar, but does not eliminate the need for foundational Python knowledge.

Add now Azure Container Apps MCP server hosting

Assessment: Preferred over App Service for MCP connector hosting. Each connector in its own container means isolated managed identity, isolated Key Vault access, and isolated scale policy. Scale-to-zero keeps idle connectors cost-free.

Recommendation: Host all MCP connectors and the governance gateway on Container Apps. Use Docker for dev/prod environment parity across the mixed Mac/Windows team.

Add in 6 months Azure AI Search Vector + keyword search / RAG

Assessment: Necessary for any use case where Claude answers questions over a document corpus rather than retrieving specific known files. Hybrid semantic + keyword search is significantly better for legal content than either approach alone — precise terminology matching plus conceptual understanding.

Recommendation: Phase 3 deployment for knowledge management and legal research use cases. Preferred over third-party vector databases (Pinecone, Weaviate) because it keeps all data within the Azure security perimeter.

Evaluate at 12 months PostgreSQL + pgvector Vector storage alternative

Assessment: A viable lighter alternative to Azure AI Search for lower-volume workloads. Deferred to avoid introducing a second database platform before Azure AI Search capability is established and its fit is proven.

Recommendation: Reassess at the 12-month governance review if Azure AI Search proves over-engineered for specific use cases.

03 Target Architecture
3.1 Full stack overview

The diagram below shows the complete target architecture, organized top-to-bottom as a trust hierarchy: users → identity → client surfaces → API/middle tier → MCP governance → MCP connectors → data. Hover over any layer for details. The colour coding reflects stack status: teal = existing, coral = new additions, purple = expanded usage, gray = governance/infrastructure.

Figure 1 — Full target architecture stack Hover layers for detail

The architecture enforces a strict top-to-bottom trust hierarchy. No data layer component is directly reachable from any client surface — everything passes through the .NET API layer or an MCP connector, both of which enforce Azure AD identity. The MCP governance gateway sits between the middle tier and every connector, meaning all connector traffic is audited regardless of which client surface initiated it.

3.2 Identity layer — Azure Active Directory

Azure AD is the single identity plane for all Claude products and MCP connectors. Three capabilities are required:

  • SSO via SAML 2.0 — all Claude products authenticate against the firm's Azure AD tenant. No separate credentials for Cowork, Claude.ai, or Code.
  • SCIM provisioning — the Anthropic Enterprise admin panel is provisioned from Azure AD group membership. Account deactivation on departure or matter roll-off propagates automatically.
  • Conditional Access — MCP connector access and Claude Code provisioning are gated by Azure AD group membership. Access is role-based and cannot be self-provisioned.
  • Ethical wall groups — attorneys subject to information barriers are excluded from corresponding iManage matter workspaces. The MCP connector inherits this by passing the user's own OAuth token rather than a service account.

3.4 MCP governance gateway

The MCP governance gateway is the most important new infrastructure component. It is a lightweight proxy — hosted on Azure Container Apps — that sits between every Claude client surface and every MCP connector. All connector traffic routes through it without exception. It performs four functions:

  1. Allowlist enforcement — only connectors explicitly listed in the organizational allowlist can be invoked. Attempts to call unlisted servers are blocked and logged.
  2. Credential management — connector credentials are retrieved from Azure Key Vault at runtime. No connector stores credentials in its container image or environment variables.
  3. Audit logging — every MCP tool call is logged with timestamp, Azure AD user identity, connector name, tool name, sanitized parameters, and response status. Forwarded to Microsoft Sentinel in real time.
  4. Prompt injection detection — tool call results are inspected for instruction-like content before being returned to Claude. Flagged results are quarantined pending review.
SECURITY
Prompt injection is a material risk for agentic systems. A malicious instruction embedded in an iManage document Claude reads could attempt to redirect Claude's actions. The gateway's injection detection and the requirement for user confirmation on irreversible actions are the primary mitigations — both must be live before any connector touches client matter data.
3.5 iManage connector — detailed flow

The iManage connector is the highest-priority build and the most sensitive from a professional responsibility standpoint. The critical design decision is user-token-passthrough: the connector passes the authenticated user's own OAuth token to iManage Cloud, not a privileged service account. Claude inherits the user's existing matter ACLs and ethical walls automatically.

4.1 CLAUDE.md — the project memory file

Every repository containing Claude-integrated code must include a CLAUDE.md file at its root. This file gives Claude Code persistent context about the repository without requiring developers to re-explain it every session. It defines: repository purpose and scope; coding standards; MCP servers in scope; iManage workspace IDs and matter data handling rules; systems explicitly off-limits; test requirements; security rules; and the deployment gating policy.

Changes to CLAUDE.md are treated as production code changes and require pull request review by the engineering lead. This makes the project's AI operating constraints visible, version-controlled, and auditable.

06 Technology Rationale
6.1 Python — why it's essential for a .NET shop

Python does not replace .NET for business logic APIs. The recommended pattern is clear: Python for AI pipeline components (document processing, embedding generation, RAG orchestration); .NET for business logic, matter services, and the internal API MCP adapter; Node.js/TypeScript for MCP connector servers.

Three reasons Python is non-negotiable for the AI pipeline layer:

  • Anthropic SDK maturity — Anthropic's Python SDK is more extensively documented, receives updates faster, and has far more community examples than the .NET SDK. Developers building against the Anthropic API find Python the path of least resistance.
  • AI/ML ecosystem — the tooling for RAG pipelines, document chunking, embedding generation, and agent orchestration exists almost exclusively in Python. Libraries like sentence-transformers, PyMuPDF, and LangChain have no meaningful .NET equivalents. Building AI pipelines in .NET means reimplementing this ecosystem from scratch.
  • Claude Code assistance — Claude Code generates high-quality Python, lowering the learning curve for developers coming from .NET or TypeScript backgrounds. The skills gap is real but bridgeable.
6.2 Node.js — natural extension of existing TypeScript investment

Node.js is the natural choice for MCP connector development: the Anthropic MCP SDK has a TypeScript-first implementation; Claude Code itself runs on Node, so tooling alignment is tight; and the team already knows TypeScript, making this an extension of existing skills rather than a new language adoption. The iManage, M365, and Azure DevOps connectors should all be Node.js/TypeScript.

6.3 Azure Container Apps — why not App Service

Container Apps is preferred over App Service for MCP connector hosting for three specific reasons:

  • Per-connector isolation — each connector runs in its own container with its own managed identity, its own Key Vault access policy, and its own scale configuration. A security issue in one connector cannot directly compromise another.
  • Scale-to-zero — most connectors will be idle most of the time. Container Apps scales to zero when idle and scales quickly on demand. App Service always-on plans would be unnecessary expense for low-traffic connectors.
  • Dev/prod parity via Docker — Container Apps runs Docker containers, meaning the same image runs on a developer's Mac or Windows machine via Docker Desktop and in production. This is especially valuable on a mixed Mac/Windows team and eliminates environment mismatch issues.
6.4 Azure AI Search — why not third-party vector databases

Azure AI Search is recommended over third-party vector databases (Pinecone, Weaviate, Qdrant) for legal document retrieval because it keeps all data within the existing Azure security perimeter, integrates with Azure AD for access control, and eliminates a separate vendor relationship and data transfer boundary.

Hybrid search — combining semantic vector search with traditional keyword search — is the right mode for legal content. Legal documents require precise terminology matching (clause numbers, defined terms, statutory citations) AND conceptual understanding (finding precedents by concept rather than exact phrase). Azure AI Search supports this hybrid mode natively; most standalone vector databases require separate keyword search infrastructure alongside them.

DECISION
Third-party vector databases are explicitly excluded from the recommended stack. They would require a new vendor relationship, data outside the Azure security perimeter, additional operational overhead, and a separate access control model — none justified by any capability gap in Azure AI Search for the firm's use cases.
07 Security Architecture Summary

Security is built into the architecture rather than layered on after the fact. The six key controls:

User token passthrough
The iManage connector passes the authenticated user's own OAuth token — not a service account. Claude inherits existing matter ACLs and ethical walls automatically. No new privilege escalation surface.
MCP governance gateway
Every connector call routes through the gateway — it cannot be bypassed. Enforces allowlist, retrieves credentials from Key Vault, logs all tool calls to Sentinel, inspects results for prompt injection.
Credential architecture
No credentials in code, container images, or environment files. All secrets in Azure Key Vault. Connectors authenticate to Key Vault using managed identity — no secrets passed between components as configuration values.
Prompt injection defense
Content Claude reads via MCP (documents, emails, API responses) is inspected before being returned to Claude. Flagged results are quarantined. Developers treat all MCP-retrieved content as untrusted input.
Audit and observability
Every MCP tool call produces a Sentinel log entry with user identity, connector, tool, and sanitized parameters. Anomalous matter access, high-volume usage, and credential anomalies trigger alerts. Monthly usage reports to AI Strategy.
CVE management
Container images scanned on build and weekly thereafter. Critical CVEs patched and redeployed within 72 hours. High CVEs within 14 days. Affected connector suspended if patch unavailable within the timeline.
