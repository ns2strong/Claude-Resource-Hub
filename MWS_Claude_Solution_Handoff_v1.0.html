01The Handoff Problem

Cowork makes it easy for a motivated attorney or legal ops specialist to build a genuinely useful solution — a matter intake workflow, a contract review pipeline, a weekly client update generator — without writing a line of code. This is a significant capability unlock. It also creates a new class of operational risk.

When that person leaves, rotates to another matter, or simply moves on to something else, the solution they built often leaves with them — or rather, it stays in production but nobody knows how it works, what it connects to, or how to change it. This is not a hypothetical: it is the standard failure mode of every previous wave of end-user computing, from Excel macros to SharePoint workflows.

Cowork-specific constraints that make this harder
  • No audit logs or compliance export (currently). Cowork activity is excluded from the Anthropic Compliance API and Data Exports — meaning you cannot retrospectively reconstruct what a solution was doing or accessing.
  • No cross-tool context sharing. Context built in a Cowork project (instructions, context files, connector config) is not visible to Claude Code or Claude.ai — making it hard for a developer to inspect or replicate a Cowork solution's logic.
  • Desktop-local session persistence. Scheduled tasks require the desktop app to stay open; context lives on the original user's machine unless explicitly extracted.
  • No native solution sharing. Cowork projects, chat history, and artifacts cannot currently be shared with other users — ownership transfer requires explicit documentation and re-creation.

The processes in this document are designed to work around these constraints — not pretend they don't exist. The core principle is that any solution which goes into production must have its logic externalised into shared, version-controlled artefacts before it gets there, not after somebody has already left.

PRINCIPLE
If a solution's logic lives only in one person's Cowork session, it is not in production — it is in a pilot. Promotion to production requires: a Solution Document in the registry, all context files in version control, connector and plugin configuration documented, and a named owner who is not the original builder.
02Solution Tiers & Classification

Solutions fall into three tiers based on how they were built. The tier determines which handoff processes apply and how much engineering involvement is required to maintain the solution. Click a scenario to expand it.

🧩
COWORK  Pure knowledge worker-built
Skills, context files, scheduled tasks, plugins — no code
Examples: Weekly matter status report generator, contract clause extraction workflow, client update email drafting, document filing automation using iManage connector, research synthesis pipeline.

Typical builder: Attorney, legal ops specialist, practice group administrator (Power User or Analyst persona).

Handoff complexity: Medium. The solution logic is in natural language (skills, context files, instructions) and is theoretically portable — but only if it was documented and externalised before the builder left. Without that, reconstruction from scratch is the only option.

Maintenance: Can usually be maintained by another Power User or Analyst persona without developer involvement, provided the documentation is adequate.
⚙️
CODE-BUILT  Developer-engineered
Claude Code, MCP connectors, APIs, Azure-deployed
Examples: iManage MCP connector, internal document processing pipeline, custom Claude-integrated React application, automated matter intake API.

Typical builder: Developer or AI Solutions Analyst (Developer persona).

Handoff complexity: Lower, if built correctly. Code-built solutions are in Azure Repos, have CLAUDE.md, and follow the standard pipeline — the handoff process is well-established standard engineering practice, augmented with Claude Code-specific artefacts.

Maintenance: Requires Developer persona. CLAUDE.md contains the institutional context needed for a new developer to pick up the project with Claude Code assistance.
🔗
HYBRID  Cowork front-end + engineered backend
Attorney-facing Cowork workflow backed by developer-built APIs or connectors
Examples: A Cowork plugin that calls a custom .NET API to retrieve matter summaries; a Cowork scheduled task that uses the iManage MCP connector plus a Python-built RAG index; a practice group Cowork workflow that depends on a developer-maintained Azure function.

Typical builders: Two people — a Power User or Analyst who built the Cowork side, and a Developer who built or maintains the backend.

Handoff complexity: Highest. Two separate handoff processes must run in parallel: the Cowork documentation and re-creation process for the knowledge worker side, and the standard engineering handoff for the backend. If one owner leaves, the other half of the solution becomes opaque.

Key risk: The Cowork owner often doesn't know the backend exists; the Developer often doesn't know what the Cowork workflow expects from the backend. Dependency mapping at build time is essential.

3.2Context file externalisation — Cowork and Hybrid solutions

The single biggest risk in Cowork solutions is context that exists only on the builder's local machine or only in their Cowork session. Every piece of context that governs solution behaviour must be externalised to a shared location before the solution goes to production.

Global instructions: The solution's system-level instructions exported to a .md file in Azure Repos
REQUIRED
Context files: All context files the solution uses committed to /context-files/[solution-id]/ in Azure Repos
REQUIRED
Skill files: Custom skill .md files committed to the MWS plugins repo and referenced by name in the Solution Document
REQUIRED
Scheduled task definitions: Natural language description of each scheduled task — what it does, its cadence, what it needs to run — committed alongside context files
REQUIRED
Connector configuration: Which connectors, which OAuth scopes, which iManage workspace IDs, documented in the Solution Document
REQUIRED
Sample task transcripts: 2–3 example sessions showing the solution working correctly — useful as test cases for successors
RECOMMENDED
Failure examples: Known failure cases and how to recover — reduces time to resolve issues for a new owner
RECOMMENDED
GATE
The AI Solutions Analyst review (Section 4.2) will not approve a Cowork solution for production registration unless all REQUIRED artefacts above are in Azure Repos and linked from the Solution Document. A solution that fails this review is classified as a pilot, not a production solution.

4.3Ownership rules
  • Every production solution must have both an owner and a deputy owner — two named, different people. A solution with a single owner is a bus-factor risk.
  • Owner and deputy owner must not both be the original builder. The deputy owner should ideally be the person most likely to inherit the solution.
  • HR integration trigger: When a named owner or deputy owner is marked for departure (offboarding process), an automated Azure DevOps work item is created for the AI Strategy & Technology team with a 30-day deadline to complete a handoff. This is the formal trigger for the handoff playbook (Section 5).
  • Quarterly review reminders: Calendar invites go to both owner and deputy owner. If a review is more than 14 days overdue, the AI Solutions Analyst team escalates to the practice group lead.
  • Practice group lead accountability: Practice group leads are accountable for ensuring all solutions in their group have valid, current owners. This is a standing item in the quarterly governance review.
05Handoff Playbook

The handoff playbook is triggered by a formal ownership change event: a departure, a role change, a practice group restructure, or an elective transfer. Three variants are described below based on whether the handoff is planned (the owner is leaving on known timeline) or unplanned (the owner has already left).

5.1Planned handoff — 30-day process

This is the ideal scenario — the owner knows they are leaving and can participate in the handoff. The 30-day timeline is structured so the outgoing owner's direct involvement is front-loaded, leaving the successor to run independently for the last week as a validation of the handoff quality.

D-30
Handoff initiated — successor identified
AI Strategy & Technology creates handoff work item. Outgoing owner and successor identified. If solution documentation is incomplete, outgoing owner prioritises completing it before any other handoff activity. Solution Document reviewed and updated.
D-25
Context file and artefact audit
Run /handoff-audit [solution-id] skill (Section 9.2). Generates a gap report: which artefacts are missing, which context files exist only on the outgoing owner's machine, which scheduled tasks would break if the desktop app is not running. All gaps resolved before D-20.
D-20
Knowledge transfer session (1–2 hours)
Outgoing owner walks successor through the solution using the Solution Document as the agenda. Session recorded (Teams). Successor runs the solution's smoke test independently with the outgoing owner present. Q&A captured in the Solution Document change history.
D-14
Successor configures solution on their environment
Successor installs required plugins, connects to MCP connectors (iManage OAuth re-authorised under successor's identity — not the outgoing owner's token), sets up scheduled tasks. Verifies each runs correctly. For Cowork solutions, context files loaded from Azure Repos, not copied from outgoing owner's machine.
D-7
Shadow period — successor runs, outgoing owner available
Successor operates the solution independently for one week. Outgoing owner available for questions but does not intervene unless there is a failure. Any issues discovered are documented and resolved. If the solution breaks during this period, the gap must be fixed before the final handoff sign-off.
D-0
Ownership transfer confirmed
Successor and AI Solutions Analyst sign off on the handoff work item. Registry updated: new owner, deputy owner. Outgoing owner's Cowork connection to iManage for this solution revoked. Old OAuth tokens rotated. Solution Document version incremented.
5.2Unplanned handoff — the owner has already left

This is the harder scenario and unfortunately the more common one. The process depends on what artefacts exist in the registry and Azure Repos. The deputy owner becomes the default interim owner immediately — which is why the requirement for a named deputy is non-negotiable.

RISK
If the departed owner's context files were never externalised to Azure Repos, and the Solution Document is incomplete or missing, reconstruction from scratch may be the only option. This is why the production registration gate exists — to prevent exactly this situation by requiring documentation before production status is granted.

Regardless of documentation state, the following must happen immediately when an owner departs without a planned handoff:

  • Deputy owner notified and confirmed as interim owner (same day)
  • Departed owner's OAuth tokens and Cowork connector access revoked (within 24 hours)
  • Solution health check run to confirm nothing is broken by the token revocation
  • If the solution uses scheduled tasks that were running on the departed owner's desktop, those tasks are suspended until the deputy owner configures them on their own environment
5.3The iManage re-authorisation requirement

This is the most important operational detail in any handoff involving iManage. Because the MWS iManage connector uses user token passthrough — meaning Claude's access to iManage is via the authenticated user's own OAuth token, not a service account — the following must always happen on ownership transfer:

  1. The old owner's OAuth token for the iManage connector is revoked as part of their offboarding. The solution stops working at this point — this is intentional and correct.
  2. The new owner (successor or deputy) re-authorises the connector under their own identity. They go through the OAuth flow as themselves, establishing their own token in the governance gateway's Key Vault.
  3. The new owner's iManage access is verified — they must have the appropriate matter workspace access under their own iManage credentials. If the departed owner had access the new owner does not have, this surfaces a matter access gap that must be resolved with iManage administration before the solution resumes.
  4. The first run after re-authorisation is monitored — the AI Solutions Analyst reviews the audit log to confirm the new owner's token is being used and the solution is accessing only the expected matters.
WHY
This re-authorisation requirement is not bureaucratic overhead — it is the mechanism that ensures Claude never has residual access to matters via a departed employee's credentials. It also surfaces matter access misalignments early: if the new owner doesn't have the access the solution needs, better to discover that in a controlled handoff than when the solution silently fails at 6am on a Monday morning.
06Extending a Solution After Handoff

Extending a solution that someone else built — especially a Cowork solution you inherited — requires a different approach than building from scratch. The successor must understand what exists before changing it, and must ensure changes don't break the parts they didn't touch.

6.1The "understand before you change" protocol

Before making any change to a solution you have inherited, run the /solution-brief [solution-id] skill against the existing Solution Document and context files. Claude will generate a structured analysis of what the solution does, why it was built the way it was (from the design rationale in the Solution Document), and what the likely consequences of different types of changes would be. This takes 10–15 minutes and prevents a large category of regression errors.

1
Read the Solution Document and run the solution
Before touching anything, run the solution end-to-end using the smoke test defined in the Solution Document. Establish a baseline — know what "working correctly" looks like before making changes.
2
Scope the change and assess impact
Use the Solution Document dependency map to identify everything the proposed change might touch. For Hybrid solutions, notify the backend developer before making changes to the Cowork side — Cowork changes can break backend assumptions about input format, connector scope, or timing.
3
Make the change in a staging context first
For Cowork solutions: duplicate the context files to a staging/ subfolder in Azure Repos; test the change there. For code-built solutions: use the standard dev → staging pipeline. Never modify a production context file directly without testing.
4
Update documentation before merging
Update the Solution Document: what changed, why, known limitations of the new version. Update context files in Azure Repos. Increment the version in the change history. The AI Solutions Analyst reviews the Solution Document update for completeness.
5
Deploy to production and run smoke test
Promote context file changes to production (merge in Azure Repos), re-run smoke test, update registry entry (last modified, version). For P1 solutions, notify the practice group lead of any material changes.

6.3Using Claude to extend a Cowork solution

Claude can be used directly to help extend a Cowork solution — ask it to analyse the existing context files, identify gaps or improvement opportunities, draft new skill instructions, or rewrite sections for clarity. The key discipline is treating this as a collaborative design process, not a one-shot prompt.

APPROACH
Load the existing context files into a Claude session, paste in the current Solution Document, and describe what you want to change. Ask Claude to: (1) explain what the existing solution does in plain English, (2) identify risks or side effects of your proposed change, (3) draft the updated context file. Review the draft against the original before committing. This is far more reliable than editing context files directly without AI assistance.
07Ongoing Maintenance
7.1Quarterly review — what to check

Every production solution receives a quarterly review regardless of whether anything has changed. The review is lightweight for stable solutions (30 minutes) and more involved if the solution has been extended, if connectors have been updated, or if personnel have changed.

Run the smoke test. Does the solution produce the expected output for the standard test input? Document the result.
ALL TIERS
Verify ownership. Are the named owner and deputy still at the firm, still in the same practice group, still the right people to own this?
ALL TIERS
Check connector health. Are all MCP connectors responding? Are OAuth tokens current? Has any connector been updated in ways that affect this solution?
ALL TIERS
Review iManage access scope. Does the solution still need access to the same workspaces? Have any matters been closed or added? Is the ethical wall configuration still correct?
MATTER DATA
Check plugin and skill versions. Are any plugins or skills used by this solution on a deprecated version? Update or document the reason for not updating.
COWORK / HYBRID
User feedback review. Has the practice group lead reported any issues or change requests? Are there known limitations that have become more painful?
RECOMMENDED
Model behaviour check. Has a recent Claude model update changed how the solution behaves? Compare output quality against the baseline from initial deployment.
RECOMMENDED
7.2Model update management

Claude model updates are one of the least visible but most impactful maintenance concerns for Cowork solutions. Unlike a code dependency where a breaking change produces a hard error, a model update can silently change output quality, format, or behaviour in ways that only a user notices — and only after the fact.

The /solution-health-check [solution-id] skill runs the solution's smoke test and compares output against the stored baseline, flagging significant divergences. This is the primary mechanism for detecting model-update-induced regressions between quarterly reviews.

08Deprecation & Retirement

Solutions that are no longer needed must be formally retired — not just abandoned. Abandoned solutions are a security and compliance liability: they retain connector access, consume quota, and may be quietly re-used by someone who doesn't know the context.

StepActionWho
1. Retirement decisionOwner or practice group lead determines solution is no longer needed. Documents rationale in Solution Document.Owner / Practice group lead
2. Dependency checkAI Solutions Analyst verifies no other solution depends on this one (check Solution Registry for cross-references).AI Solutions Analyst
3. Revoke connector accessRemove solution's connector authorisations from governance gateway. Revoke any OAuth tokens specific to this solution. Verify via audit log.IT Security
4. Deactivate scheduled tasksAll scheduled tasks for this solution deactivated on the owner's desktop and any shared infrastructure.Owner
5. Archive artefactsSolution Document, context files, and code moved to /archive/[solution-id]/ in Azure Repos. Not deleted — retained for 3 years per records management policy.AI Solutions Analyst

IMPORTANT
A solution that fails two consecutive quarterly reviews without resolution (smoke test fails and not fixed within 30 days) is automatically flagged for retirement review. The AI Solutions Analyst initiates a retirement discussion with the practice group lead. If no one advocates for maintaining the solution within 14 days of that discussion, it is retired.

# Solution Document Generator

Interview the user to collect all required fields for a Solution Document.
Ask questions in conversational groups — do not present as a form.

## Phase 1: Identity and purpose (5 mins)
- What does this solution do? Describe it as if explaining to a partner.
- Who uses it, and how often?
- What did they do before this solution existed?
- Which practice group owns this work?

## Phase 2: Technical inventory (10 mins)
- Which MCP connectors does it use? (iManage, M365, internal APIs?)
- If iManage: which workspaces does it access? What data does it read/write?
- What plugins and skills does it use?
- Does it have any scheduled tasks? What do they do and when do they run?
- Where do outputs go? Who receives them?

## Phase 3: Handoff readiness (5 mins)
- Where are your context files? Are they in Azure Repos?
- Who would you want to take over if you left tomorrow?
- What would they need to know that isn't written down anywhere?
- What are the most common failure modes and how do you fix them?

Generate:
1. Complete Solution Document in the standard template
2. Artefact gap list (what still needs to be committed to Azure Repos)
3. Azure DevOps work item body for solution registration submission
4. Suggested deputy owner from the practice group roster if none named
