01 Executive Summary

Purpose. This standard defines how McDermott Will & Schulte deploys, governs, and secures the Anthropic Claude AI platform — including Claude Cowork, Claude Code, and custom MCP connectors — across the firm's technology environment. It establishes access control requirements, security architecture standards, data handling obligations, and operational governance for all Claude-related systems.

Why this matters. Claude Cowork and Claude Code are agentic AI tools: they can autonomously read, write, and act on documents and systems on behalf of users. Unlike passive AI chatbots, these tools can access iManage matter workspaces, send messages, file documents, and execute multi-step workflows. Governance appropriate for a chatbot is insufficient for an agentic platform.

Relationship to existing firm AI policy. McDermott's firm-wide AI Use Policy governs what attorneys and staff may use AI for. This document governs how the Claude platform is built, secured, and operated. The two documents are complementary and should be read together by anyone deploying or administering Claude-integrated systems.

PRINCIPLE 1
Identity-first
Every Claude action is traceable to an authenticated user via Azure Active Directory. No service accounts or shared credentials are used for matter-sensitive operations.
PRINCIPLE 2
Least privilege
Users access only the systems and data their role and matter assignments require. MCP connector access is provisioned by Azure AD group, not self-service.
PRINCIPLE 3
Audit by default
All MCP tool calls — including which matters were accessed, by whom, and when — are logged to the firm's security infrastructure and retained per the firm's records management policy.
PRINCIPLE 4
Legal privilege first
The iManage connector and all document-handling plugins respect existing iManage ACLs and ethical wall configurations. Claude cannot access a matter a user does not already have access to.
GC / RISK
This document requires sign-off from Risk Management and the General Counsel's office before production deployment of any MCP connector that touches client matter data. Sections 3 and 4 contain the controls most relevant to professional responsibility and client confidentiality obligations.
02 Access Control and Persona Model
2.1 Governing principle

Access to Claude products and MCP connectors is provisioned through Azure Active Directory group membership. No user may self-provision access to Claude Code, API keys, MCP connectors, or Cowork plugins beyond their assigned persona tier. Requests for elevated access follow the standard IT change request process and require manager approval.

2.3 Access provisioning rules
  • Azure AD group membership is the sole provisioning mechanism. Manual access grants outside this process are prohibited.
  • SCIM provisioning from Azure AD to the Anthropic Enterprise admin panel ensures that account deactivation propagates automatically, typically within one provisioning cycle.
  • Ethical wall enforcement: attorneys subject to an iManage ethical wall are automatically excluded from the corresponding matter workspace. The MCP connector inherits this restriction by passing the user's own OAuth token — not a service account — to iManage Cloud.
  • Plugin marketplace access in Cowork is controlled via the admin Customize panel. Practice group plugins are visible only to the corresponding Azure AD group. Attorneys cannot install arbitrary plugins from outside the firm marketplace.
2.4 Privileged access — developers and analysts
  • API keys are issued per-developer, never shared, and rotated on a 90-day cycle or on personnel change, whichever is sooner.
  • API keys are stored in Azure Key Vault. They are never stored in source code, environment files committed to Azure Repos, or shared via Teams or email.
  • Claude Code access on developer workstations is provisioned via the mwe-claude-developers Azure AD group. Workstations must be MDM-enrolled and meet the firm's endpoint security policy.
  • AI Solutions Analysts receive Claude Code access scoped to non-production environments. Production MCP connector deployment requires developer persona.
  • All API usage is logged and attributed to the issuing key. Anomalous usage patterns trigger a Sentinel alert.
03 Security Architecture and MCP Governance
3.1 MCP governance gateway

All MCP connector traffic routes through a governance gateway hosted on Azure Container Apps. The gateway is not optional and cannot be bypassed. It performs four functions:

  • Allowlist enforcement: only connectors explicitly listed in the organizational allowlist may be invoked. Attempts to invoke unlisted MCP servers are blocked and logged.
  • Credential management: connector credentials are retrieved from Azure Key Vault at runtime. No connector stores credentials locally or in its container image.
  • Audit logging: every MCP tool call is logged with timestamp, user identity (Azure AD object ID), connector name, tool name, input parameters (sanitized), and response status. Logs forwarded to Microsoft Sentinel.
  • Prompt injection detection: the gateway inspects tool call results for instruction-like content before returning them to Claude. Flagged results are quarantined and reviewed before any action is taken.
SECURITY
Prompt injection is a material risk for agentic AI systems. Malicious content embedded in a document Claude reads from iManage could instruct Claude to take unintended actions — filing a document to the wrong matter, forwarding content externally. The gateway's injection detection and the requirement for user confirmation on irreversible actions are the primary mitigations.

3.3 Cowork desktop app controls
  • The Cowork Chrome extension is restricted to an allowlisted set of domains, managed via MDM policy and reviewed quarterly. Attorneys may not add domains to the allowlist.
  • File system access granted to Cowork is scoped to designated working folders only.
  • Scheduled tasks created in Cowork must be reviewed and approved by a manager before activation. Unreviewed scheduled tasks are disabled after 48 hours.
3.4 Claude Code controls
  • Claude Code is provisioned only to MDM-enrolled developer workstations meeting the firm's endpoint security policy.
  • CLAUDE.md files in each repository define the scope of Claude Code's operation. Changes to CLAUDE.md require the same review process as production code changes.
  • Claude Code subagents committed to .claude/agents/ are treated as production code and subject to the full code review and pipeline process.
  • Claude Code must not be used to access, process, or transmit client data from outside the firm's managed environment.
  • MCP servers available to Claude Code in each project are explicitly declared in CLAUDE.md and restricted to connectors approved for that project.
04 Data Handling, Confidentiality, and GDPR
4.1 Client confidentiality
  • Client matter data must not be processed by Claude in ways inconsistent with the firm's engagement terms or applicable confidentiality rules.
  • Attorneys are responsible for ensuring that matter content submitted to Claude is appropriate to share with an AI system under the client's engagement terms.
  • Claude output incorporating client matter content — summaries, drafts, analysis — must be reviewed by a qualified attorney before external use. Claude output is never final work product without attorney review.
  • The iManage connector's user-token-passthrough architecture ensures Claude cannot access matter documents beyond what the authenticated user is permitted to see.
RPC 1.6
Attorneys must exercise professional judgment about whether submitting specific matter content to an AI system is consistent with their duty of confidentiality. This platform standard provides the technical controls to support that obligation — it does not alter it.

4.3 Prohibited uses

The following uses are prohibited under this standard, irrespective of the firm-wide AI Use Policy:

  • Processing client data on non-MDM-enrolled or personal devices.
  • Sharing API keys, OAuth tokens, or MCP connector credentials outside the approved credential management process.
  • Using Claude to draft, send, or file external communications without attorney review.
  • Configuring MCP connectors to access systems not on the organizational allowlist.
  • Using Claude Code or the Anthropic API outside the firm's managed Azure environment for work involving client matter data.
  • Bypassing or disabling audit logging for any Claude operation.
  • Sharing Claude output containing client matter content with third parties without appropriate confidentiality protections.

06 Delivery Pipeline and Change Management
6.1 Claude Code development standards
  • Every repository must contain a CLAUDE.md file defining: repository purpose, coding standards, MCP servers in scope, iManage workspace IDs, systems off-limits, test requirements, security rules, and deploy gating policy.
  • Changes to CLAUDE.md are treated as production code changes and require pull request review by the Engineering lead.
  • Subagents committed to .claude/agents/ are treated as production code.
  • Claude Code operates in Plan mode by default for non-trivial changes. Claude proposes; engineers decide.
6.2 Azure DevOps pipeline requirements

Six pipeline stages are required for all Claude-integrated application repositories:

  1. PR review — a Claude Code PR review subagent runs automatically on every pull request, flagging security issues, data handling concerns, and CLAUDE.md deviations.
  2. Build and test — standard Azure Pipelines build with automated test suite. Claude Code test generation subagent maintains coverage for new MCP tool implementations.
  3. Security scan — container image scanning and static analysis on every build. Critical findings block promotion.
  4. Pre-deploy gate — Claude Code subagent validates MCP connector configurations against the allowlist and verifies audit logging is functional before promoting to staging.
  5. Staging validation — all deployments promote to staging before production, using non-production iManage workspaces and sandboxed M365 tenancy.
  6. Production release — gated release requiring Engineering lead approval. No direct-to-production deployments permitted.
6.3 MCP connector deployment process

New MCP connectors must complete the following process before production deployment:

  • Connector specification document reviewed by IT Security and Engineering lead
  • Code review completed with security checklist (Section 3.2)
  • Penetration assessment for connectors touching client matter data
  • Connector added to organizational allowlist in the MCP governance gateway
  • Staging validation including audit log verification
  • Risk Management notification for connectors accessing matter data
  • Production deployment via Azure DevOps gated release pipeline

A Glossary
TermDefinition
Claude CoworkAnthropic's desktop AI productivity platform for knowledge workers. Supports multi-step agentic tasks, file access, MCP connectors, and a private plugin marketplace.
Claude CodeAnthropic's agentic coding tool operating in the terminal. Used by developers and AI solutions analysts for software development, MCP connector building, and pipeline automation.
MCP (Model Context Protocol)An open standard that allows Claude to connect to and act on external systems via a defined tool interface. Connectors expose named tools Claude can invoke.
MCP connectorA software component implementing the MCP server specification for a specific external system (iManage, M365, etc.). Each connector exposes a set of named tools.
MCP governance gatewayThe firm's centralized proxy through which all MCP connector traffic routes. Enforces allowlists, manages credentials, logs tool calls, and detects prompt injection.
CLAUDE.mdA project-level file giving Claude Code persistent context about a repository: purpose, standards, MCP servers in scope, and operational constraints.
SubagentA specialized Claude Code agent committed to .claude/agents/ in a repository. Performs a specific recurring task automatically in the CI/CD pipeline.
Ethical wallAn information barrier in iManage preventing specified users from accessing specific matter workspaces. Inherited by the MCP connector via user token passthrough.
Prompt injectionAn attack where malicious instructions in content Claude reads attempt to redirect Claude's actions. The governance gateway detects and quarantines suspicious results.
SCIMSystem for Cross-domain Identity Management. Used to automatically provision and deprovision Claude accounts based on Azure AD group membership.
RAGRetrieval-Augmented Generation. Claude retrieves relevant documents from a search index before generating a response, grounding answers in actual firm content.
User token passthroughThe iManage connector design pattern where the user's own OAuth token (not a service account) is passed to iManage, so Claude inherits the user's existing access controls.
