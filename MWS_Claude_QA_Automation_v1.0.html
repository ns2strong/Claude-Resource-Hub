01The AI Testing Challenge

Testing Claude-built solutions requires a fundamentally different mindset from testing traditional software. The challenge is non-determinism: the same input to a traditional function always produces the same output — that's what makes unit tests reliable. An LLM-powered solution, given the same input, may produce outputs that differ in wording, structure, or nuance across runs. Both may be correct. Neither will match expected_output exactly.

This means the traditional assertion-based testing model is insufficient on its own. It works well for the deterministic parts of a solution (API calls, data transformations, connector behaviour), but it cannot evaluate whether a Claude-generated legal summary is accurate, whether a contract review surfaces the right risks, or whether a matter update email maintains the appropriate professional tone.

What traditional tests can evaluate
  • Does the API return the correct status code?
  • Is the connector passing the user's OAuth token?
  • Does the iManage search return results in under 2s?
  • Is the output JSON schema valid?
  • Does the hook correctly block a credential pattern?
What requires LLM-graded evaluation
  • Is the contract summary accurate and complete?
  • Does the output follow MWS brand voice?
  • Did the solution identify the material risks?
  • Is the attorney disclaimer present and appropriate?
  • Did the agent take the correct sequence of steps?
CORE RULE
Use deterministic graders wherever possible. Use LLM graders where deterministic is insufficient. Use human review for high-stakes validation. Deterministic graders are fast, cheap, and reliable. LLM graders are flexible and scalable but add cost and latency. Human review is the highest quality bar but the most expensive — reserve it for initial calibration and P1 solution acceptance.

A second challenge specific to MWS is the legal domain. Solutions handling client matter data must be tested not just for technical correctness, but for legal appropriateness: privilege preservation, ethical wall enforcement, attorney disclaimer requirements, and RPC 1.6 compliance. These require a dedicated evaluation layer that standard engineering test frameworks do not provide.

3.1LLM-as-judge — how to write good rubrics

LLM graders are only as good as their rubrics. Vague rubrics produce inconsistent verdicts. Specific, measurable rubrics produce reliable, scalable evaluation. The following principles apply to every LLM grader written for MWS solutions:

  • Emit a verdict first, reasoning second. Ask the judge to output only PASS or FAIL (or a 1–5 score) before any explanation. This forces a concrete judgment and makes the output parseable by downstream automation.
  • One rubric per criterion. Don't ask the judge to evaluate accuracy, tone, and disclaimer presence in a single prompt. Use three separate judge invocations — one per criterion — and aggregate results. This makes failures traceable to specific criteria.
  • Make criteria binary where possible. "Does the output mention the attorney review disclaimer? YES or NO" is more reliable than "Rate the appropriateness of the disclaimer on a scale of 1–5."
  • Calibrate with human-reviewed examples. Before deploying a rubric, collect 20–30 human-reviewed outputs (10 clearly good, 10 clearly bad, 10 borderline). Verify the judge agrees with human verdicts on the clear cases before trusting it on borderline ones.
  • Version your rubrics. Rubrics stored in Azure Repos alongside the test suite. When a rubric changes, run both old and new against the calibration set before deploying to CI.
Example: MWS LLM judge pattern (TypeScript)
// judges/brand-voice.ts
export async function judgesBrandVoice(output: string): Promise<JudgeResult> {
  const response = await anthropic.messages.create({
    model: 'claude-sonnet-4-6',
    max_tokens: 256,
    system: `You are a strict brand compliance judge for McDermott Will & Schulte.
      Evaluate the provided text against MWS brand standards.
      Respond ONLY with: PASS or FAIL, then a newline, then one sentence explaining why.`,
    messages: [{
      role: 'user',
      content: `Evaluate this output for MWS brand voice compliance:

CRITERIA (ALL must pass):
1. No prohibited phrases: revolutionary, game-changing, cutting-edge, unleash, supercharge
2. No legacy firm name "McDermott Will & Emery" (should be "McDermott Will & Schulte")
3. No anthropomorphic Claude descriptions (thinking, feeling, believing)
4. Attorney review disclaimer present if output contains legal analysis

OUTPUT TO EVALUATE:
${output}`
    }]
  });
  const text = response.content[0].text;
  const verdict = text.startsWith('PASS') ? 'PASS' : 'FAIL';
  const reason = text.split('\n').slice(1).join(' ').trim();
  return { verdict, reason, criterion: 'brand-voice' };
}
3.2Behaviour-based testing for agent solutions

For solutions where Claude takes a sequence of actions (multi-step Cowork workflows, agent pipelines), the most important question is not "is the final output correct?" but "did the agent take the correct sequence of steps?" This is behaviour-based quality testing — evaluating the trace, not just the terminus.

The pattern: run the solution, capture the tool call trace, apply deterministic judges to observable facts (which tools were called, in what order, with what parameters), and LLM judges to the quality of each step's output. This gives you both structural correctness (the agent did the right things) and quality assurance (it did them well).

Trace-based judge pattern
// For a solution that should: search_matter → get_document → summarise → file_document
const traceJudges = [
  // Deterministic: did the right tools fire?
  { type: 'deterministic', check: trace =>
    ['search_matter','get_document','file_document']
      .every(t => trace.toolCalls.includes(t))
  },
  // Deterministic: did user token passthrough occur? (audit log check)
  { type: 'deterministic', check: trace =>
    trace.auditLog.every(e => e.tokenType === 'user')
  },
  // LLM: was the summary accurate and complete?
  { type: 'llm', criterion: 'summary-quality',
    prompt: summaryQualityRubric
  },
  // LLM: was the attorney disclaimer present in any output containing analysis?
  { type: 'llm', criterion: 'disclaimer-present',
    prompt: disclaimerRubric
  },
];

05Layer 2 — AI Output Evaluation

AI output evals are the most distinctive part of the MWS testing framework. They evaluate the quality, accuracy, and compliance of Claude-generated content — the things traditional tests cannot assess. Every solution that produces output consumed by attorneys or sent externally must have an eval suite.

5.1Standard eval rubrics — required for all AI-output solutions

The following rubrics are applied to all solutions that produce text output. They are packaged in the mws-standard-evals module and applied automatically by the eval-runner subagent. Solutions may add domain-specific rubrics on top of these standards.

LLM-GRADED Brand voice compliance
Criteria (all binary PASS/FAIL):
  • No prohibited phrases (revolutionary, game-changing, cutting-edge, unleash, supercharge, leverage as verb, paradigm shift, seamlessly)
  • Firm name correct: "McDermott Will & Schulte" (not legacy "McDermott Will & Emery")
  • Claude not described anthropomorphically (thinking, feeling, believing, understanding)
  • Active voice used in >70% of sentences
  • Second person ("you", "your") used in attorney-facing copy
Pass threshold: All 5 criteria must pass. Any FAIL = overall FAIL.
Calibration set: 30 examples in /evals/calibration/brand-voice/
LLM-GRADED Attorney review disclaimer
Applies to: Any output containing legal analysis, contract review, matter summarisation, or risk assessment.

Criteria:
  • A disclaimer stating attorney review is required before external use is present in the output
  • The disclaimer is prominent — not buried in a footnote
  • The output does not characterise itself as "advice", "opinion", or "counsel"
Pass threshold: All 3 criteria must pass. This is a non-negotiable requirement — FAIL blocks deployment.
Note: This rubric has a deterministic pre-check: grep for "attorney" case-insensitively. If not found, skip LLM judge and immediately FAIL.
LLM-GRADED Output completeness and accuracy
Applies to: Solutions that summarise, extract, or analyse specific content (contract review, matter update, research synthesis).

Criteria (scored 1–5):
  • Completeness: Does the output address all aspects of the task defined in the solution's context file? (5 = fully, 1 = major gaps)
  • Accuracy: Are factual claims consistent with the source material? (5 = fully consistent, 1 = material inaccuracies)
  • Hallucination: Does the output introduce information not present in the source? (5 = none detected, 1 = material fabrication)
Pass threshold: Average score ≥ 4.0. Any individual criterion below 3 = immediate FAIL.
Calibration: Requires 20 human-reviewed examples per solution before the rubric is considered validated.
DETERMINISTIC Output format and schema
Applies to: Solutions that produce structured output (JSON, tables, specific section headings).

Checks:
  • JSON outputs validate against the declared schema (Zod / JSON Schema)
  • Required sections present (for document-format outputs, all expected headings exist)
  • Output length within defined bounds (prevents runaway or truncated outputs)
  • No raw error text or stack traces in output (indicates the solution encountered an unhandled error)
Pass threshold: All checks must pass — binary, no partial credit.
5.2Regression baseline management

AI output quality can regress silently when a Claude model is updated or when context files are changed. The regression baseline captures a set of canonical input/output pairs at deployment time. Subsequent eval runs compare current outputs against the baseline using LLM judges to detect quality drift.

  • Baseline captured at: Initial production deployment, after any major model update, after any material context file change.
  • Baseline contents: 20–50 input examples with human-reviewed "gold standard" outputs, stored in /evals/baselines/[solution-id]/ in Azure Repos.
  • Drift detection: The eval-runner subagent compares current outputs to the baseline weekly. A score drop of >10% on the completeness/accuracy rubric triggers an alert to the solution owner.
  • Model update protocol: When Anthropic releases a new model, run the full eval suite against both current and new model before updating. Promote only if new model scores ≥ current on all rubrics.

7.2Accessibility testing — WCAG 2.1 AA

Every UI surface is tested against WCAG 2.1 AA using a combination of automated (axe-core via Playwright) and Claude-assisted evaluation. Automated axe-core catches structural violations; the accessibility-check subagent evaluates contextual requirements that axe-core misses — error message quality, loading state descriptions, and colour-independent status indicators.

8.3Container Apps scale-to-zero and cold start testing

Azure Container Apps scale-to-zero is a cost advantage, but cold starts introduce latency that must be characterised and within acceptable bounds. For each MCP connector:

  • Cold start latency measured: from first request to a scaled-to-zero instance to response received. Maximum acceptable: 8 seconds.
  • Warm start latency: measured after the container is warm. Maximum acceptable: connector-specific threshold from Section 8.1.
  • Scale-up behaviour: verify the connector scales to handle the baseline profile within 30 seconds of load beginning.
  • Key Vault secret retrieval on cold start: verify credentials are retrieved correctly on startup and are not stale.

9.3Human acceptance testing — when and how

Human acceptance testing is not UAT in the traditional sense — it is specifically targeted at the things automated tests cannot adequately cover. It is required for P1 solutions and any solution that a qualified attorney will rely on for client-facing work.

  • Initial LLM rubric calibration: Before any LLM judge is trusted, an attorney or senior analyst reviews 20–30 outputs and labels them PASS/FAIL. The LLM judge is not deployed until it agrees with human labels on ≥ 90% of the calibration set.
  • P1 acceptance sign-off: A named attorney from the relevant practice group runs the solution on 5–10 real scenarios (using test matter data) and signs off that outputs meet professional quality standards before production deployment.
  • Post-model-update review: When a major model update changes output quality beyond the 10% drift threshold, a small human review (5 examples) confirms the regression is real before a fix is prioritised.

13Rollout Sequence — What to Build When

Build the testing infrastructure in the same phased sequence as the platform itself — each layer of tests supports the platform layer it validates.

P1
Phase 1 — Foundation (now, alongside platform Phase 1)
Unit test patterns for MCP connectors. SAST (Semgrep) and secret scan (GitLeaks) in CI pipeline. Brand voice LLM judge + attorney disclaimer judge — the two highest-value, lowest-cost evals. Calibration process documented and first 30-example calibration sets created. test-gen and eval-runner subagents deployed.

P2
Phase 2 — iManage connector and pipeline (alongside platform Phase 2)
Full iManage connector unit test suite (5 required properties). Integration tests in staging against sandboxed iManage workspaces. Ethical wall enforcement tests. Audit log to Sentinel validation. Playwright E2E for any UI surfaces in the first connector-backed applications. axe-core accessibility scanning in CI.
DETERMINISTICLLM-GRADED
P3
Phase 3 — Scale and compliance (alongside platform Phase 3)
k6 load testing for iManage connector and M365 connector. Baseline load profiles established. Legal compliance automated checks. security-audit subagent in pre-production gate. Output completeness / accuracy eval rubrics calibrated by practice group attorneys for first Cowork solutions going to P1 production. Human acceptance process formalised.

P4
Phase 4 — Full automation and practice-specific evals (alongside platform Phase 4)
Regression baseline management automated. Model update comparison workflow. Practice-specific eval rubrics for M&A, Health Regulatory, and other practice groups. Soak testing for all shared connectors. Performance monitoring dashboards in Azure Monitor. eval-calibrate skill available to AI Solutions Analysts for self-service rubric creation.

KEY INSIGHT
The two highest-ROI investments in this entire framework are: (1) the brand voice + attorney disclaimer LLM judges in Phase 1 — they catch the most frequent and most professionally consequential output quality issues with minimal setup cost; and (2) the iManage connector unit tests in Phase 2 — they validate the most security-sensitive infrastructure component before any attorney data touches it. Build these first, build everything else on top.
