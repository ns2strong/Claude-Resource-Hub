01 Claude Code Extension Model — Primer

Claude Code has six distinct extension points. Each solves a different problem. The enterprise delivery system uses all six, layered so that governance controls are automatic and non-bypassable, quality standards are encoded once and applied everywhere, and developers spend their time on business logic rather than ceremony.

Extension typeWhat it isHow it activatesPrimary use at MWS
SKILL Markdown playbook that teaches Claude a repeatable workflow. Invoked as a slash command or auto-detected from context. Manual /skill-name OR automatic when Claude matches context to description Coding standards, branding checks, document templates, architecture patterns, MCP connector scaffolding
HOOK Shell script that fires deterministically at a lifecycle event — before/after a tool call, session start/end, pre-commit. Does not involve AI reasoning. Automatic, event-driven, always fires — cannot be bypassed by Claude Lint enforcement, credential scanning, security gates, brand validation, audit logging, test runners
SUBAGENT Isolated Claude instance with its own context window, tool permissions, and instructions. Runs in parallel or sequentially as a specialist. Spawned by lead Claude agent when task matches specialist description, or explicitly invoked PR review, security audit, test generation, documentation, compliance check — heavy tasks that would pollute main context
AGENT TEAM Coordinated group of agents with defined roles. Lead agent delegates to specialists. Configured via prompt and optional teams.json. Triggered by lead agent; teammates inherit permissions with optional fine-tuning via hooks Full feature delivery pipeline — architect + developer + tester + security auditor running in parallel on complex deliverables
PLUGIN Packaged bundle of skills, hooks, subagents, and MCP connections. Installable from the firm marketplace. Versioned and governed. All components activate automatically on session start after installation MWS standard dev toolchain, practice-group packs, MCP connector bundles — distributed via private GitHub as plugin source
MCP External system connection via Model Context Protocol. Gives Claude tools to read from and act on iManage, M365, DevOps, databases. Available as tools in any session where the MCP server is declared in project config iManage document access, Azure DevOps context, M365, internal APIs — all routing through the MWS governance gateway
KEY DISTINCTION
Hooks are the only extension type that is deterministic and non-bypassable. Skills and subagents can be ignored if Claude doesn't recognise the context. Hooks always fire at their event trigger regardless of what Claude is doing. This makes them the right tool for mandatory governance controls — credential scanning, lint gates, security validation. Skills are the right tool for repeatable workflows that Claude applies with judgment. Choose deliberately.

3.1 SKILL  solution-brief — structured intake

Invoked as /solution-brief at the start of any new application or feature. Claude asks structured questions and generates a brief that auto-populates the project's CLAUDE.md skeleton, ensuring every project begins with the governance-critical fields pre-filled rather than discovered later.

description: Generate a structured MWS solution brief for a new application or feature
disable-model-invocation: true
---

# MWS Solution Brief Generator

Gather the following from the user through conversation, then produce:
1. A completed solution brief document
2. A pre-populated CLAUDE.md skeleton for the new repository

## Required information to collect
- Business problem and intended users (attorney, legal ops, developer?)
- Matter data involved? (yes/no + sensitivity level)
- Deployment target: SharePoint portal OR Azure Container Apps agentic workload?
- MCP connectors required (iManage, M365, internal APIs, DevOps?)
- Estimated user volume and availability requirements
- Practice group owner and IT Security contact

## Hosting decision rule (enforce this)
- Portal/dashboard with no agent background tasks → SharePoint/SPFx
- Any of: streaming responses, background agents, MCP server hosting,
  WebSocket connections → Azure Container Apps. Do NOT suggest SharePoint.

## Output: CLAUDE.md skeleton
Populate with collected values:
- repo purpose, MCP servers in scope, iManage workspace IDs if applicable
- what NOT to modify (matter ID mapping, privilege tagging, ethical wall logic)
- test requirements, security rules, deploy gating policy

Auto-invoked whenever Claude is working on anything involving matters, clients, documents, or legal workflows. Injects standing context about privilege, confidentiality, and ethical walls so Claude's design suggestions are legally-aware by default — not as an afterthought.

user_invocable: false

# MWS Legal Domain Context

When designing or building features involving legal workflows, always apply:

## Attorney-client privilege
- Matter content accessed via MCP is privileged by default
- No caching matter content outside approved storage (iManage, Azure Blob)
- Claude output summarizing matter content requires attorney review before external use

## Ethical walls
- Never aggregate matter data across ethical wall boundaries
- User token passthrough is mandatory — inherit iManage ACLs, never bypass
- Scheduled tasks that access matter data must re-validate access at runtime

## Data minimisation
- Request only the document fields needed for the specific operation
- Do not log matter content in application logs — log metadata (matter ID, doc ID) only
- RAG indexes over matter content require separate privilege classification per chunk
04 Stage 2 — Development
4.1 HOOK  lint-on-write — immediate standards enforcement

Fires after every file write, unconditionally. Runs the appropriate linter for the file type. Claude sees the output and self-corrects immediately rather than accumulating lint debt across a session. This is placed as a hook rather than a skill because it must run every single time without exception.

settings.json — PostToolUse hook configuration
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "bash .claude/hooks/lint-on-write.sh $CLAUDE_TOOL_INPUT_PATH"
          }
        ]
      }
    ]
  }
}
.claude/hooks/lint-on-write.sh
#!/bin/bash
FILE=$1
EXT="${FILE##*.}"

case $EXT in
  ts|tsx|js|jsx)
    npx eslint --fix "$FILE" && npx prettier --write "$FILE"
    ;;
  py)
    python -m black "$FILE" && python -m ruff check --fix "$FILE"
    ;;
  cs)
    dotnet format --include "$FILE" 2>/dev/null || true
    ;;
  md)
    # Run brand voice check hook for markdown content files
    .claude/hooks/brand-voice-check.sh "$FILE"
    ;;
esac
4.2 HOOK  credential-scan — mandatory secret detection

Runs on every file write alongside the linter. Uses a pattern library tuned for the MWS environment — Anthropic API keys, Azure connection strings, iManage OAuth tokens, SQL Server credentials. Exits non-zero on detection, blocking Claude from proceeding and surfacing the finding immediately.

.claude/hooks/credential-scan.sh
#!/bin/bash
FILE=$1
FINDINGS=""

# Anthropic API key
grep -n "sk-ant-" "$FILE" && FINDINGS="anthropic-key"

# Azure connection strings
grep -in "DefaultEndpointsProtocol=https" "$FILE" && FINDINGS="$FINDINGS azure-connstring"

# iManage OAuth tokens (bearer pattern)
grep -in "Bearer [A-Za-z0-9._-]\{40,\}" "$FILE" && FINDINGS="$FINDINGS imanage-token"

# SQL Server credentials
grep -in "Password=[^;]" "$FILE" && FINDINGS="$FINDINGS sql-password"

# Generic high-entropy strings in likely credential contexts
grep -inE "(api_key|secret|token|password)\s*[=:]\s*['\"][A-Za-z0-9+/]{20,}" \
     "$FILE" && FINDINGS="$FINDINGS generic-secret"

if [ -n "$FINDINGS" ]; then
  echo "CREDENTIAL SCAN FAILED: $FILE"
  echo "Potential secrets detected: $FINDINGS"
  echo "All credentials must be stored in Azure Key Vault."
  echo "Use managed identity or Key Vault references instead."
  exit 1
fi
4.3 HOOK  pre-commit-gate — last line before git

The final deterministic gate before any commit reaches Azure Repos. Blocks sensitive file extensions, enforces commit message format (required for Azure DevOps work item linking), and re-runs credential scan on the staged diff. This is defence-in-depth — the PostToolUse credential scan should have caught secrets earlier, but this hook is the backstop.

.claude/hooks/pre-commit.sh
#!/bin/bash
# Block sensitive file extensions
SENSITIVE=$(git diff --cached --name-only | grep -E '\.(env|key|pem|p12|pfx|creds)$|^\.env')
if [ -n "$SENSITIVE" ]; then
  echo "BLOCKED: Attempting to commit sensitive files:"
  echo "$SENSITIVE"
  exit 1
fi

# Enforce commit message format: [TYPE] AB#1234 Description
# TYPE: feat|fix|chore|docs|test|refactor|security
MSG=$(git log --format="%s" -1 HEAD 2>/dev/null || cat .git/COMMIT_EDITMSG 2>/dev/null)
if ! echo "$MSG" | grep -qE "^\[(feat|fix|chore|docs|test|refactor|security)\] (AB#[0-9]+|NO-TICKET) .+"; then
  echo "BLOCKED: Commit message format required:"
  echo "  [TYPE] AB#1234 Description"
  echo "  [feat] AB#4521 Add iManage search_matter MCP tool"
  exit 1
fi

# Final credential scan on staged diff
git diff --cached | grep -E "^\+" | grep -iE \
  "(sk-ant-|DefaultEndpointsProtocol|Bearer [A-Za-z0-9]{40}|Password=[^;])" \
  && echo "BLOCKED: Credential pattern in staged diff" && exit 1

echo "Pre-commit gate: PASSED"
4.4 SKILL  mws-coding-standards — auto-applied style guide

Auto-invoked on every coding session. Gives Claude persistent knowledge of MWS-specific standards beyond what generic linters enforce: naming conventions, preferred patterns, error handling requirements, logging standards, and what is explicitly prohibited. This is the primary mechanism for encoding institutional knowledge into every Claude Code session.

.claude/skills/mws-coding-standards/SKILL.md (excerpt)
---
name: mws-coding-standards
description: Apply MWS coding standards when writing or modifying any code file.
user_invocable: false
---

# MWS Coding Standards

## TypeScript / Node.js (MCP connectors)
- All MCP tool implementations must be in separate files under src/tools/
- Tool names: snake_case (search_matter, get_document, file_document)
- Every tool must declare input schema with Zod validation
- Error handling: never throw raw errors — return typed MCP error responses
- Logging: structured JSON to stdout, never console.log in production code
- Async patterns: prefer async/await; no unhandled promise rejections

## .NET (C# — business logic APIs)
- Controller actions return IActionResult with explicit HTTP status codes
- All matter-sensitive operations require [MatterAuthorize] attribute
- Connection strings via IConfiguration — never hardcoded
- Use CancellationToken in all async controller actions
- Entity Framework migrations require review before applying to production schema

## Python (AI pipelines)
- Type hints required on all function signatures
- Pydantic models for all data structures passed between pipeline stages
- Azure Key Vault via DefaultAzureCredential — no manual credential passing
- All LLM calls wrapped in retry logic with exponential backoff
- Document chunk metadata must include: matter_id, doc_id, chunk_index, sensitivity_level

## PROHIBITED in all languages
- No hardcoded matter IDs, workspace IDs, or user identifiers
- No console.log / print() in production code paths
- No direct SQL queries outside the .NET data access layer
- No synchronous HTTP calls that could block event loops

05 Stage 3 — Testing & Validation
PRINCIPLE
Testing at MWS has two distinct failure modes: code that doesn't work, and code that works but handles matter data incorrectly. The technical test suite catches the first. The legal compliance check and mcp-connector-test catch the second. Both are required before any application touches production matter data.
5.1 HOOK  test-runner — continuous test execution

Runs the relevant test suite automatically after every code write. Scoped to tests related to the changed files — not the full suite on every keystroke. Claude sees test failures immediately and self-corrects within the same task, dramatically reducing the test-fix cycle time.

.claude/hooks/test-runner.sh
#!/bin/bash
FILE=$1
EXT="${FILE##*.}"
DIR=$(dirname "$FILE")

case $EXT in
  ts|tsx)
    # Run tests related to changed file
    npx jest --testPathPattern="${FILE%.*}" --passWithNoTests 2>&1 | tail -20
    ;;
  py)
    # Find and run related pytest tests
    TESTFILE="${FILE/src\//tests/}"
    TESTFILE="${TESTFILE/.py/_test.py}"
    [ -f "$TESTFILE" ] && python -m pytest "$TESTFILE" -v --tb=short 2>&1 | tail -20
    ;;
  cs)
    # Run dotnet tests in the same project
    TESTPROJ=$(find . -name "*.Tests.csproj" | head -1)
    [ -n "$TESTPROJ" ] && dotnet test "$TESTPROJ" --no-build 2>&1 | tail -20
    ;;
esac

6.2 SKILL  deploy — gated deployment execution

The deploy skill is the only mechanism for production deployment. It is explicitly configured with disable-model-invocation: true — Claude cannot trigger it automatically. Only a developer explicitly typing /deploy [environment] initiates a deployment. The skill validates the pre-deploy gate passed, triggers the Azure DevOps pipeline, monitors progress, and runs post-deploy verification.

MANDATORY
disable-model-invocation: true on the deploy skill is non-negotiable. An agentic system that can deploy itself to production without explicit human invocation is a security and governance failure. No automated pipeline path should ever invoke /deploy — deployments are always human-initiated.
6.3 HOOK  post-deploy-verify — health confirmation

Fires on the Stop event after a deploy task completes. Runs health checks, a smoke test against the deployed application, and verifies audit logging is live in the production environment. Posts result to the Azure DevOps deployment record and alerts the engineering lead on failure.

.claude/hooks/post-deploy-verify.sh (triggers on Stop after /deploy)
#!/bin/bash
TARGET_URL=$1
ENV=$2

# Health check
HTTP_STATUS=$(curl -s -o /dev/null -w "%{http_code}" "$TARGET_URL/health")
if [ "$HTTP_STATUS" != "200" ]; then
  echo "HEALTH CHECK FAILED: $TARGET_URL returned $HTTP_STATUS"
  # Post failure to Azure DevOps pipeline
  az devops build tag add --build-id $BUILD_ID --tags "deploy-health-fail"
  exit 1
fi

# Verify audit logging is live (check Sentinel for recent entries)
RECENT_LOGS=$(az monitor log-analytics query \
  --workspace $SENTINEL_WORKSPACE \
  --analytics-query "MWSMCPAudit | where TimeGenerated > ago(5m) | count" \
  --output tsv 2>/dev/null)

echo "Post-deploy health: OK | Audit log entries (last 5m): $RECENT_LOGS"
echo "Deployment to $ENV: VERIFIED"

# MWS Brand Standards

## Firm name
Always use full form on first reference: "McDermott Will & Schulte"
Short form thereafter: "the firm" or "MWS" (not "McDermott" alone)
Never: "McDermott Will & Emery" (legacy name — merger completed)

## Voice
- Authoritative but accessible — no legalese in UI copy
- Active voice preferred; passive voice only when agent is unknown
- Present tense for instructions; past tense for completed actions
- Second person ("you", "your matter") for attorney-facing interfaces

## Prohibited phrases (AI-generated clichés)
Never use: "revolutionary", "game-changing", "cutting-edge", "next-level",
"unlock", "unleash", "supercharge", "leverage" (as verb), "paradigm shift",
"seamlessly", "robust solution", "AI-powered" (too generic — be specific)

## Prohibited in Claude output descriptions
Never describe Claude as: "thinking", "feeling", "believing", "understanding"
Use instead: "identifies", "generates", "suggests", "processes", "retrieves"

## Accessibility
- All UI copy must work as screen reader text (no icon-only buttons)
- Error messages must state what went wrong AND what to do next
- Loading states must describe what is happening ("Retrieving matter documents...")
- Colour cannot be the sole distinguisher for status (add text/icon)

## Legal-specific copy rules
- Never describe Claude output as "advice", "opinion", or "counsel"
- Always include: "Review by a qualified attorney is required before external use"
  on any UI surface that displays Claude-generated matter analysis
8.2 HOOK  brand-voice-check — automated enforcement

Hooks complement the branding skill with deterministic pattern matching. The skill teaches Claude the standards; the hook enforces them as a hard gate on markdown content files. Catches the legacy firm name in particular — a category of error that will otherwise silently persist across documents for years.

.claude/hooks/brand-voice-check.sh
#!/bin/bash
FILE=$1
VIOLATIONS=""

# Legacy firm name — most critical catch
grep -in "McDermott Will.*Emery" "$FILE" \
  && VIOLATIONS="$VIOLATIONS legacy-firm-name"

# Prohibited AI clichés in user-facing content
grep -inE "\b(revolutionary|game.changing|cutting.edge|unleash|supercharge)\b" \
  "$FILE" && VIOLATIONS="$VIOLATIONS ai-cliche"

# Anthropomorphic Claude descriptions
grep -inE "Claude (thinks|feels|believes|understands|knows)" \
  "$FILE" && VIOLATIONS="$VIOLATIONS anthropomorphic"

# Missing attorney review disclaimer on analysis content
if echo "$FILE" | grep -qE "analysis|summary|review|memo"; then
  grep -qiE "attorney.*review|qualified.*attorney" "$FILE" \
    || VIOLATIONS="$VIOLATIONS missing-attorney-disclaimer"
fi

if [ -n "$VIOLATIONS" ]; then
  echo "BRAND CHECK FAILED: $VIOLATIONS in $FILE"
  exit 1
fi

11.3 Priority resolution — enterprise vs project vs plugin

When skill names conflict across levels, Claude Code resolves in this order: enterprise > personal > project > plugin. MWS uses this deliberately: enterprise-level overrides ensure firm standards cannot be quietly overridden by a project-level skill with the same name. Plugin skills use the plugin-name:skill-name namespace to avoid all conflicts.

CAUTION
Never create a project-level skill named mws-coding-standards or mws-branding. These names are reserved for the enterprise-level versions that override everything. A project-level skill with the same name would be silently superseded and never invoked, giving developers a false sense that their customisation is active.
12 Build & Rollout Sequence

Don't try to build everything at once. The sequence below layers extensions in dependency order — each tier is usable independently before the next is built. The first six items deliver the most value with the lowest complexity and should be operational by the end of Phase 2.

Build first — Phase 1 / 2 (highest ROI)
Foundational governance hooks + mws-standard-dev plugin
credential-scan, lint-on-write, pre-commit-gate hooks + mws-coding-standards skill + pr-review subagent. These eliminate the most common categories of rework and security issues immediately. Package as mws-standard-dev plugin and push to all developers via admin auto-install. No dependencies on connectors or other infrastructure — buildable on day one.
Build second — Phase 2 (iManage connector milestone)
iManage-specific stack + mws-imanage-connector plugin
legal-domain-context skill + mcp-connector-scaffold skill + mcp-connector-test subagent + legal-compliance-check subagent. Bundle as mws-imanage-connector plugin. These are required before the iManage MCP connector goes to production — the test and compliance subagents are the go/no-go gate for matter data access.
Build third — Phase 2 / 3 (deployment automation)
Deployment and operations stack
pre-deploy-gate subagent + deploy skill (disable-model-invocation) + post-deploy-verify hook + solution-brief skill. Once the connector pattern is established and developers are using the standard plugin, these automate the deployment ceremony and ensure every project starts with a complete brief and CLAUDE.md.
Build fourth — Phase 3 (security hardening)

Build fifth — Phase 4 (full automation + practice packs)
Agent teams + practice packs + operations skills
